\section{Discussion and Limitations}
\label{sec:discussion}

\subsection{Interpretation of the placement result}
\label{sec:disc_tradeoff}

The useful result is asymmetry, not an aggregate ``GPU speedup.''  On Thor, CPU/OpenSSL fits the AES-GCM data path, while the GPU helps independent ML-KEM batches and measured envelope refresh.  AEGIS-Q therefore exposes an elastic maintenance lane; current telemetry can throttle mounted FUSE writes but still does not justify foreground AI-QoS recovery.

\subsection{Security limits}
\label{sec:disc_energy}

The per-file envelope is authenticated and block records are AEAD-protected, but the mount password remains the root credential.  The implementation has no hardware-backed credential release, audited rotation protocol, privileged-attacker defense, or side-channel evaluation.  File anchors are replayable.  TPM mode requires an administrator-provisioned index; retained probes cover fail-closed replay-after-advance and transient PCR-policy behavior, not persistent PCR-bound key release under provisioned TPM policy.

\subsection{Evaluation limits and roadmap}
\label{sec:disc_roadmap}

The three-run primitive measurements cannot establish steady-state throughput, tail latency, energy efficiency, or portability.  The frozen-contract matrix provides warm-cache plaintext, gocryptfs, and AEGIS-Q rows, but it is not yet a complete fscrypt, dm-crypt, fs-verity, dm-integrity, OP-TEE, SPDK, or GPUDirect Storage comparison.  Historical fscrypt and dm-crypt fio outputs are retained only for traceability, not current comparison evidence, because they do not sweep the same request sizes, cache states, queue depths, sync modes, and application workloads.

The POSIX envelope is also narrow: ordinary read/write/\texttt{fsync} and narrow disjoint writers are evaluated, while shared \texttt{mmap}, rename, directory \texttt{fsync}, and arbitrary conflicting writes are not deployment claims.

The crash story has the same boundary.  The repository retains a conservative crash-audit map, a daemon \texttt{SIGKILL} cutpoint matrix, a SQLite selected-boundary oracle campaign, SQLite/\texttt{dbm.dumb} same-store TPM replay-after-advance runs, and SQLite syscall-exact app-crash timing, but it still lacks physical power-loss, kernel-crash, drive-cache, and evidence across broader workloads.  Stronger claims still need kernel-encryption baseline rows, exact I/O/cache/clock controls, foreground p99 telemetry control, power-fail injection with a provisioned TPM policy, a non-NVIDIA UMA platform for CUDA-independent deployment, and side-channel testing if claimed.

\subsection{Deployment takeaway}

The evaluated deployment envelope is a local edge service whose foreground SQLite database shares mounted secure storage with elastic background writes.  This case study is the same \texttt{sqlite-mounted-qos-recovery-2026-06-27} contract used by Figure~\ref{fig:first_page_qos} and Table~\ref{tab:qos_sqlite_recovery}, not a separate anecdotal demo.  The useful property is policy separation plus TPM replay-after-advance fail-closed behavior; it does not extend to kernel-encryption replacement, TensorRT p99 recovery, unmanaged power loss, or CUDA-independent deployment.

\subsection{CUDA-specific boundary}
\label{sec:disc_portability}

The implementation depends on CUDA managed memory and cuPQC.  It uses \texttt{cudaMemPrefetchAsync} and stream synchronization as executor-local locality and ordering operations, not storage-DMA registration or permissions.  Porting the policy requires a separate proof of the target API's allocation, synchronization, and device-I/O semantics.
