\section{Related Work}
\label{sec:related_work}
\label{sec:related_sim}

AEGIS-Q separates storage protection, GPU/storage placement, and PQC key-plane work.  fscrypt and dm-crypt are mature kernel baselines~\cite{fscrypt, dmcrypt}, and gocryptfs is the closest user-space encrypted-filesystem comparison point~\cite{gocryptfs}; FUSE overhead is a cost boundary, not a free kernel substitute.  fs-verity and dm-integrity validate integrity-only boundaries, while AEGIS-Q combines authenticated FUSE records, storage-visible policy, optional GPU maintenance, and external replay checks.

The comparison is asymmetric rather than deleted: plaintext/lowerfs, gocryptfs, dm-crypt, and AEGIS-Q are mode-aligned measured rows; fscrypt is unavailable with proof, not an implied speedup.  These direct same-contract rows expose publication and QoS costs, not kernel-replacement maturity; work that requires a different kernel, GPU file API, driver path, or security boundary remains related work rather than an artificial throughput row.

GPU/storage systems validate different contracts.  FlashNeuron and Fastensor stage large GPU workloads~\cite{flashneuron, fastensor}; SnuQS manages out-of-core capacity~\cite{snuqs_storage}, while SnuQS alone is not filesystem-format evidence.  Speculative GPU encryption, SPDK, GPUDirect Storage, and fscrypt-GPU optimize masks, movement, GPU access, or throughput, but not AEGIS-Q's FUSE records, xattrs, external anchors, D/J/C publication, TPM replay, or storage-visible QoS contract~\cite{spec_gpu_cfs, flashneuron, fscrypt_gpu, fastensor}.  GPUstore, GPUfs, GeminiFS, and GPU4FS expose storage services or filesystem acceleration to GPU-side execution~\cite{gpustore,gpufs,geminifs,gpu4fs}; GPUDirect Storage exposes a direct storage/GPU data path through cuFile~\cite{gpudirect_storage}.  AEGIS-Q keeps sidecar I/O and asks when a mounted encryption runtime should admit optional GPU key-plane work without changing the authenticated format.  OP-TEE/dm-crypt use different boundaries~\cite{optee, dmcrypt}; CUDA/cuPQC evidence is platform-specific.

Linux storage QoS mechanisms such as blk-throttle, \texttt{io.latency}/\texttt{IO.cost}, BFQ, and CFQ operate below file-encryption semantics; AEGIS-Q's \texttt{ionice}/\texttt{IOWeight} rows are controls, not an exhaustive scheduler study.  F2FS, ScaleFS, FastCommit, encrypted databases, integrity trees, and secure logs place append/replay oracles at other boundaries; AEGIS-Q's analogue is a scoped FUSE sidecar epoch log and bounded leader/follower barrier.

ML-KEM establishes shared secrets rather than per-block bulk encryption~\cite{mlkem}; post-quantum encrypted-file-system cost models reinforce that primitive choice is not a mounted runtime~\cite{pppqefs}.  AEGIS-Q keeps AES-GCM in the data plane and uses ML-KEM only for batched key-plane work.  OP-TEE/TrustZone isolates attested code paths~\cite{optee}; AEGIS-Q does not provide that TEE boundary, and stronger TPM rollback claims still need explicit NV authorization, key-release policy, and recovery semantics.  It is not a replacement for kernel encryption, POSIX filesystems, TEEs, or GPU-storage stacks, but a secure edge-storage runtime.
